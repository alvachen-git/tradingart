# AI 协作开发规范 v1.0（TradingArt）

本文件约束本仓库内所有 AI 编程助手行为。  
核心目标：**新功能开发要小步、安全、可验证**。

## 1. 三条硬性规则（必须执行）
- 分区修改：开发新功能时，必须分区小步修改，禁止一次性大范围改动。
- 功能自测：每次开发新功能后，AI 必须自行跑一遍相关测试或最小可运行验证。
- 严重错误沉淀：若出现严重错误，必须先写入经验文件 `经验记录.md`，再继续后续开发或交付。

## 2. 分区修改定义
- 一个“分区”是一个独立的小目标（例如：数据层、业务逻辑层、UI层中的一个）。
- 每次只改一个分区，完成后先验证，再进入下一个分区。
- 禁止把“数据处理 + 接口改造 + UI 重构”放在同一次大改里。
- 如需跨多个文件，必须说明每个文件的改动目的。

## 3. 新功能标准流程（强制）
1. 需求重述：用一句话说明本次功能目标和边界。
2. 分区计划：先列出本次要改的分区（A/B/C）与对应文件。
3. 分区实施：按分区逐个修改，不跳步。
4. 分区验证：每完成一个分区，进行一次最小验证。
5. 功能总验收：新功能完成后，必须再跑一遍端到端或关键路径自测。
6. 结果汇报：明确写出“改动内容、测试命令、测试结果、已知风险”。

## 4. 代码质量底线
- 只修改与当前功能直接相关的代码。
- 不引入与任务无关的依赖。
- 不提交任何密钥、token、密码等敏感信息。
- 新逻辑必须有基本异常处理，外部请求必须有超时与失败兜底。
- 避免重复代码，重复逻辑优先抽取函数。

## 5. 提交前检查清单（必须全部为是）
- [ ] 是否按分区小步完成，而不是一次性大改
- [ ] 是否对每个分区都做了最小验证
- [ ] 是否对新功能做了至少一次完整自测
- [ ] 是否记录了测试命令与结果
- [ ] 是否确认无敏感信息泄漏
- [ ] 是否评估了主要回归风险

## 6. 输出模板（统一）
AI 每次任务完成后按以下结构回复：

```text
需求目标：
- ...

分区修改：
- 分区A（文件...）：...
- 分区B（文件...）：...

测试验证：
- 命令：...
- 结果：...

风险与后续：
- 风险：...
- 后续建议：...
```

## 7. 持续改进机制
- 每次任务结束补充 1 条“本次最易错点”和 1 条“下次预防动作”。
- 同类错误出现 2 次及以上，必须在本文件新增防复发规则。
- 防复发规则格式：`触发条件 -> 强制动作 -> 验证方式`。

## 8. 严重错误经验沉淀（强制）
- 触发条件：出现严重错误（如功能不可用、关键数据错误、任务中断、用户侧明显异常）。
- 记录位置：`经验记录.md`。
- 严重级别：按“第 9 节 严重级别判定标准（S1/S2）”执行。
- 最低记录内容：错误现象、影响范围、根因、修复动作、防复发动作、验证方式。
- 门禁要求：未完成经验记录，不得标记任务完成。

## 9. 严重级别判定标准（S1/S2）
- S1：主流程不可用、关键数据错误、交易相关结论明显错误、数据污染/丢失、安全风险。
- S2：非主流程异常、局部功能不可用但有替代路径、性能明显下降但系统可运行。
- 无法快速判断时，按 S1 处理（从严）。
- S1 处理要求：立即记录到 `经验记录.md`，优先修复，修复并验证前不继续新功能开发。
- S2 处理要求：在当前任务结束前完成记录并给出防复发动作。

## 10. 防复发规则（初始）
- 改数据处理逻辑 -> 必测空值/缺失字段 -> 用最小样例运行验证。
- 改脚本入口或定时任务 -> 必查路径与环境变量 -> 先 dry-run 再正式执行。
- 改 UI 字段 -> 必核对字段存在和类型 -> 打开页面人工检查一次。
- 改手动布局算法（含 `free_w`/锚点/扇形） -> 必做固定窗口下“元素数量递减”与高 DPI 场景回归 -> 提供前后截图或明确观测结论。

## 11. 常用命令（按需）
- 安装依赖：`pip install -r requirements.txt`
- 启动应用：`streamlit run Home.py`
- 示例测试：`python test_options_data.py`

冲突优先级：**系统指令 > 用户指令 > 本文件**。
