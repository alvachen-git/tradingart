### 记录 39（S2）地图开战前预清理逻辑运行时类型转换报错，导致战斗无法开启
- 时间：2026-02-24
- 错误现象：
  - 重启后测试“地图开战 -> ESC 测试退出 -> 再次地图开战”时，无法开启战斗。
  - Godot 终端报错：`Invalid call. Nonexistent 'int' constructor.`，定位到 `map.gd::_preclean_stale_linked_battle()`。
  - 因开战前预清理函数提前异常退出，地图入口后续只能停留在原状态或回退到异常流程。
- 影响范围：
  - 地图入口“进入卡牌战斗测试”路径（尤其在 `godot_local` 模式下）被阻断，无法继续验证本地 SQLite 抽样供数。
- 根因：
  - `map.gd` 新增预清理逻辑直接对接口返回 `Variant` 使用 `int(...)` 转换，在该运行时路径触发 GDScript 运行时构造调用异常（编译门禁无法提前捕获）。
  - 同函数内后续若继续使用 `int(...)` 解析接口返回字段，也存在相同风险。
- 修复动作：
  - 在地图开战链路引入本地稳健整数解析函数（不依赖 `int(...)` 构造调用），统一解析 `linked_battle_run_id`、`battle_run_id` 等接口字段。
  - 补充日志，明确预清理读取到的 `linked_battle_run_id / battle_state`，便于后续排查。
- 防复发动作：
  - GDScript 中处理接口返回 `Variant` 数值字段时 -> 禁止在关键主流程直接使用裸 `int(...)` 构造转换 -> 统一走本地 `safe_int` 解析函数（支持 int/float/string/Nil）并做一次真实路径手测。
- 验证方式：
  - 地图入口开战不再出现 `Nonexistent 'int' constructor` 报错。
  - ESC 测试退出后再次地图开战，前端能成功进入新战斗或输出明确后端错误，而不是在预清理阶段崩溃。

### 记录 46（S1）期权工具 SQL 前缀查询 `%` 未转义导致 LangSmith 工具报错并误导交易结论
- 时间：2026-02-25
- 错误现象：
  - 询问“碳酸锂期权”相关问题时，LangSmith 中 `get_commodity_iv_info`、`check_option_expiry_status` 等工具返回 `unsupported format character '\'' (0x27)`。
  - 上层 Agent 因工具失败/空结果，输出中误判为“无标准化场内期权”或类似结论。
- 影响范围：
  - 商品期权相关工具链（IV、到期日、部分市场快照）中所有依赖 `sql_prefix_condition()` 的 SQL 查询。
  - 可能影响交易判断与策略建议准确性，属于交易相关结论明显错误风险。
- 根因：
  - `symbol_match.py::sql_prefix_condition()` 返回 `LIKE 'LC%'`（单 `%`）。
  - 在 MySQL 驱动执行 SQL 时，底层会按 Python `%` 规则处理 SQL 字符串，导致 `%` 后紧跟 `'` 被当成非法格式占位符，抛出 `unsupported format character`。
- 修复动作：
  - 将 `sql_prefix_condition()` 的 `LIKE` 模式改为 `LIKE 'LC%%'`（双 `%`），兼容 DBAPI 的 `%` 格式化行为。
  - 增加最小回归验证，覆盖单字母（REGEXP）与多字母（LIKE）前缀场景。
- 防复发动作：
  - 新增/修改 SQL 片段生成函数时 -> 若可能经过 DBAPI 执行链路 -> `LIKE` 中 `%` 必须统一使用 `%%` -> 用最小字符串格式化复现（模拟驱动 `%` 处理）验证不报错。
- 验证方式：
  - 运行最小复现脚本：对含 `sql_prefix_condition('LC')` 的 SQL 做 `% {}` 格式化不再报错。
  - 回归 `get_commodity_iv_info('碳酸锂')`、`check_option_expiry_status('碳酸锂')` 工具，确认不再出现 `unsupported format character`。

### 记录 47（S1）切换 qwen3.5-plus 后 ChatTongyi 调用 400 InvalidParameter 导致 AI 主流程不可用
- 时间：2026-02-25
- 错误现象：
  - 将底层模型从 `qwen-plus` 改为 `qwen3.5-plus` 后，AI 问答主流程在 LangGraph 执行阶段报错并中断。
  - 终端与 LangSmith 显示 DashScope 返回 `status_code: 400`, `code: InvalidParameter`, `message: url error, please check url!`。
  - 前端页面提示“分析过程中出现错误，请稍后重试”。
- 影响范围：
  - 主问答链路（含 `tasks.py` 异步处理和首页 AI 分析）不可用，属于主流程不可用（S1）。
- 根因：
  - 已确认当前运行环境 `.venv311` 使用 `dashscope==1.25.1`，其 SDK 源码内仅覆盖 `qwen-turbo/qwen-plus/qwen-max` 等旧模型，未包含 `qwen3.5-plus`。
  - 当前 `langchain_community.ChatTongyi` + `dashscope==1.25.1` 组合对 `qwen3.5-plus` 请求走到不兼容 URL/endpoint 路径，DashScope 返回 `400 InvalidParameter: url error`。
- 修复动作：
  - 检查 `ChatTongyi` 版本与实现逻辑，确认当前环境对 `qwen3.5-plus` 不兼容。
  - 先将主流程与报告脚本默认模型回退为 `qwen-plus`，恢复系统可用性。
  - 保留后续升级 SDK 后再切换的验证路径（先探针、后灰度）。
- 防复发动作：
  - 升级模型名 -> 先做最小真实调用探针（单条 `ChatTongyi.invoke("ping")`）-> 再改主流程默认模型 -> 在日志中记录模型与成功/失败结果。
- 验证方式：
  - 主流程问答恢复可用，无 `InvalidParameter/url error`。
  - LangSmith 中 `ChatTongyi` 节点显示所用模型与响应正常。
